# Отчет о тестировании параллельного умножения матриц с использованием OpenMP

## 1. Цель работы
Разработка программы на C++ для умножения квадратных матриц, реализация последовательной и параллельной версий алгоритма с использованием директив OpenMP, а также исследование эффективности параллелизации в зависимости от размера матрицы и количества потоков.

## 4. Анализ результатов

1.  **Корректность вычислений:**
    Программа успешно распараллелена с помощью OpenMP. Время выполнения последовательной версии (`1 поток`) растет пропорционально кубу размера матрицы (O(N³)), что соответствует теоретической вычислительной сложности (Таблица 6).

2.  **Эффективность параллелизации:**
    *   Наилучшая эффективность достигнута при использовании **2 потоков (95.5%)**.
    *   При использовании **4 потоков** эффективность остается высокой **(82.5%)**.
    *   При переходе к **8 потокам** эффективность резко падает до **49.5%** (Таблица 4). Это связано с ростом накладных расходов на создание потоков и синхронизацию доступа к памяти, которые начинают преобладать над выигрышем от вычислений.

3.  **Производительность:**
    *   Максимальная пиковая производительность **(4249.6 MFLOPS)** достигнута на матрице **800×800** с использованием **8 потоков**.
    *   Для больших матриц (1200+ с 8 потоками) производительность стабилизируется на уровне ~4100-4200 MFLOPS, что говорит о достижении предела пропускной способности памяти или влиянии архитектурных ограничений тестового стенда (Таблица 2).

4.  **Масштабируемость:**
    *   Наибольший относительный прирост производительности наблюдается при переходе от последовательной версии к параллельной с **2 потоками (+91% ускорения)**.
    *   Переход с 4 на 8 потоков дает наименьший прирост **(+0.66 к ускорению)** из-за описанных выше накладных расходов (Таблица 7).

## 5. Заключение
В ходе работы была успешно реализована и протестирована параллельная версия алгоритма умножения матриц. Экспериментально подтверждена эффективность использования OpenMP для данной задачи. Оптимальным соотношением производительности и эффективности для данного тестового окружения является использование **4 потоков**. Дальнейшее увеличение числа потоков (до 8) нецелесообразно из-за падения эффективности ниже 50%, хотя и дает некоторый прирост абсолютной производительности.
